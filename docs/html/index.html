Vue d'ensemble

RunFlow est une plateforme self-hosted d'automatisation et d'exécution de jobs orientée API. Elle orchestre le cycle complet : déclenchement, file d'attente, exécution distribuée et collecte des résultats.

Trigger→ Job→ Queue→ Worker→ Runner Docker→ Résultat

Stack technique

  • API — Python 3.13, FastAPI, PostgreSQL 17, Valkey
  • Worker — Python asyncio, Docker Engine API
  • Web — Next.js, Monaco Editor
  • SDK — from runflow import args, result

Modes de déploiement

  • Local docker-compose.yml — tout-en-un
  • Production deploy/docker-compose.server.yml — serveur Traefik
  • Agents deploy/docker-compose.agents.yml — workers distants
Ouvrir ce guide : double-cliquez sur docs/html/index.html dans votre navigateur, ou servez-le via un serveur web statique. Aucune connexion Internet requise.

Architecture

┌─────────────────────────────────────────────────────────────┐ │ SERVEUR (Traefik / proxy) │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────┐ │ │ │ Web │ │ API │ │ Postgres │ │ Valkey │ │ │ │ Next.js │──▶│ FastAPI │──▶│ (DB) │ │ (queue) │ │ │ │ :3000 │ │ :8000 │ └──────────┘ └─────────┘ │ │ └──────────┘ └────┬─────┘ │ └──────────────────────┼────────────────────────────────────────┘ │ HTTPS (long polling) ┌─────────────┼─────────────┐ ▼ ▼ ▼ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Agent 1 │ │ Agent 2 │ │ Agent N │ │ Worker │ │ Worker │ │ Worker │ │ Docker │ │ Docker │ │ Docker │ └──────────┘ └──────────┘ └──────────┘

Principes clés

  • Pull-based — les workers demandent les runs (long polling), le serveur ne les contacte jamais.
  • Queue Postgres — assignation atomique via FOR UPDATE SKIP LOCKED.
  • Isolation Docker — chaque job s'exécute dans un conteneur éphémère sans accès au socket Docker.
  • Bind mounts — toutes les données persistent sur l'hôte (data/), pas de volumes Docker nommés.

Installation — développement local

Pour tester rapidement sur votre machine avec tous les services sur un seul hôte.

Étapes

# 1. Cloner et configurer
git clone <repo-url> RunFlow && cd RunFlow
cp .env.example .env

# 2. Lancer (crée data/ et build les runners)
./scripts/setup.sh

# 3. Créer l'administrateur
docker compose exec api runflow create-admin --email admin@runflow.local

# 4. (Optionnel) Job de démonstration
docker compose exec api runflow-seed seed-demo-job

# 5. Accéder à l'interface
open http://localhost:3000   # Web
open http://localhost:8000/docs  # API Swagger

Installation — serveur production (Traefik)

Stack serveur : API + Web + Postgres + Valkey. Pas de worker sur ce serveur.

Checklist de mise en place

  1. Créer le réseau Traefik : docker network create proxy
  2. Configurer les DNS (A records) vers le serveur
  3. Copier et éditer le fichier .env
  4. Créer les répertoires de données
  5. Builder et démarrer la stack
  6. Créer le compte administrateur
  7. Vérifier l'accès HTTPS web + API

Commandes

# Depuis la racine du dépôt
cp deploy/.env.server.example .env

# Éditer .env — exemple :
#   RUNFLOW_WEB_HOST=runflow.mondomaine.com
#   RUNFLOW_API_HOST=api.runflow.mondomaine.com
#   CORS_ORIGINS=https://runflow.mondomaine.com
#   NEXT_PUBLIC_API_URL=https://api.runflow.mondomaine.com
#   JWT_SECRET=<secret-aléatoire-long>
#   RUNFLOW_MASTER_KEY=<clé-32-octets-minimum>
#   POSTGRES_PASSWORD=<mot-de-passe-fort>

mkdir -p data/postgres data/runflow

docker compose -f deploy/docker-compose.server.yml up -d --build

# Créer l'admin (mot de passe demandé interactivement)
docker compose -f deploy/docker-compose.server.yml exec api \
  runflow create-admin --email admin@votredomaine.com

Labels Traefik configurés

ServiceRoutePort interne
WebHost(`RUNFLOW_WEB_HOST`)3000
APIHost(`RUNFLOW_API_HOST`)8000
Important — rebuild web : NEXT_PUBLIC_API_URL est injectée au build de l'image web. Si vous changez le domaine API, relancez docker compose -f deploy/docker-compose.server.yml up -d --build web.

Récupérer l'ID organisation (pour les workers)

# Se connecter à Postgres
docker compose -f deploy/docker-compose.server.yml exec postgres \
  psql -U runflow -d runflow -c "SELECT id, name, slug FROM organizations;"

Installation — agents workers

Déployez un agent sur chaque machine capable d'exécuter Docker. Les agents se connectent à l'API publique en HTTPS.

Checklist par agent

  1. Cloner le dépôt (ou copier les fichiers deploy)
  2. Configurer .env agent
  3. Construire les images runner
  4. Créer un token d'inscription côté serveur
  5. Enregistrer l'agent (une seule fois)
  6. Démarrer le worker
  7. Vérifier le statut « online » dans l'UI Workers

Étape 1 — Configuration agent

cp deploy/.env.agents.example .env

# Éditer :
#   RUNFLOW_API_URL=https://api.runflow.mondomaine.com
#   WORKER_NAME=geneva-agent1

Étape 2 — Images runner

./deploy/build-runners.sh
# Construit :
#   runflow/runner-python:0.1.0
#   runflow/runner-bash:0.1.0
#   runflow/runner-ansible:0.1.0

Étape 3 — Token d'inscription (côté serveur)

docker compose -f deploy/docker-compose.server.yml exec api \
  runflow worker-create-registration-token \
  --name geneva-agent1 --org-id <ORG_ID>

# Affiche : rf_reg_... (affiché UNE SEULE FOIS)

Étape 4 — Enregistrement (côté agent)

mkdir -p data/worker-geneva-agent1

REGISTRATION_TOKEN=rf_reg_VOTRE_TOKEN \
  docker compose -f deploy/docker-compose.agents.yml --profile register run --rm register

# Le token permanent rf_wkr_... est sauvé dans :
#   data/worker-geneva-agent1/worker.env

Étape 5 — Démarrage

docker compose -f deploy/docker-compose.agents.yml up -d worker

# Vérifier les logs
docker compose -f deploy/docker-compose.agents.yml logs -f worker

Données agent

data/
└── worker-geneva-agent1/
    ├── worker.env          # Token permanent (rf_wkr_...)
    └── pip-cache/          # Cache pip persistant

Premier démarrage

1. Connexion à l'interface

Ouvrez https://<RUNFLOW_WEB_HOST> (ou http://localhost:3000 en local). Connectez-vous avec l'email et le mot de passe admin créés via CLI.

2. Créer un job Python

  1. Menu Jobs → Créer un job
  2. Nom : Mon premier job, slug : mon-premier-job, runner : python
  3. Onglet Code → créer main.py :
from runflow import args, result

name = args.get("name", "monde")
print(f"Hello, {name}!")
result.set({"greeting": f"Hello, {name}!"})

3. Ajouter un paramètre

Onglet Parameters → ajouter name (type string, requis).

4. Lancer le job

Onglet Run → renseigner name → Exécuter. Consultez le résultat JSON et les logs.

5. Vérifier le worker

Menu Workers — le worker doit apparaître en statut online. Si absent, vérifiez les logs agent et le token.

Jobs

Un job est une unité d'exécution avec un type de runner, des fichiers source et un schéma de paramètres.

Types de runners

RunnerUsage
pythonScripts Python avec SDK RunFlow
bashScripts shell
ansiblePlaybooks Ansible (nécessite runner + inventaire)

Sources

  • Internal — fichiers édités dans l'UI (Monaco), stockés sous data/runflow/jobs/
  • Git — synchronisation depuis un dépôt au moment du claim (Phase 3)

Paramètres supportés

string, integer, float, boolean, select, multi_select, secret, json, file, date, datetime, email, url, ip, cidr, raw

Injection des arguments (dans le conteneur)

  • RUNFLOW_ARGS_FILE=/runflow/input/args.json
  • RUNFLOW_ARG_<NOM> — variables d'environnement

Parsers de résultat

ParserComportement
runflow_sdkLit le fichier résultat du SDK (recommandé)
json_stdoutParse la sortie stdout comme JSON
last_json_lineDernière ligne JSON de stdout
nonePas de résultat structuré

Exécutions (Runs)

Cycle de vie

queued→ assigned→ preparing→ running→ success / failed / timeout / cancelled

Lancer via API

# Asynchrone (202)
curl -X POST "https://api.example.com/api/v1/jobs/mon-job/run" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"arguments": {"name": "RunFlow"}}'

# Synchrone (attend la fin)
curl -X POST "https://api.example.com/api/v1/jobs/mon-job/run?wait=true&wait_timeout=120" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"arguments": {"name": "RunFlow"}}'

Logs en temps réel (SSE)

curl -N "https://api.example.com/api/v1/runs/<RUN_ID>/logs/stream" \
  -H "Authorization: Bearer $TOKEN"

SDK Python

Le SDK est pré-installé dans l'image runner Python.

from runflow import args, result

# Lire les arguments
domain = args["domain"]
verbose = args.get("verbose", False)

# Logique métier
print(f"Traitement de {domain}")

# Résultat structuré (JSON)
result.set({
    "domain": domain,
    "status": "ok",
    "checked_at": "2026-07-11T12:00:00Z"
})

Installation locale (développement)

cd packages/python-sdk
uv sync
# ou : pip install -e packages/python-sdk

Triggers

TypeDéclenchement

Webhook avec HMAC

# Créer le trigger
curl -X POST https://api.example.com/api/v1/triggers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Deploy hook",
    "trigger_type": "webhook",
    "target_id": "<JOB_ID>",
    "config": {"auth": "hmac", "secret": "mon-secret"}
  }'

# Déclencher (calculer HMAC-SHA256 du body)
curl -X POST https://api.example.com/api/v1/hooks/<HOOK_TOKEN> \
  -H "Content-Type: application/json" \
  -H "X-RunFlow-Signature: sha256=<HMAC>" \
  -d '{"branch": "main"}'

Schedule (cron)

{
  "name": "Sauvegarde nocturne",
  "trigger_type": "schedule",
  "target_id": "<JOB_ID>",
  "config": {
    "cron": "0 2 * * *",
    "timezone": "Europe/Paris"
  }
}

Modes d'auth webhook : none, bearer, secret_header, hmac.

Workflows

Orchestration de jobs en DAG avec dépendances, conditions et mapping d'arguments Jinja2.

Concepts

  • Workflow — définition (nom, slug, politique d'échec)
  • Node — référence un job + mapping + condition optionnelle
  • Edge — dépendance entre nœuds
  • WorkflowRun — instance d'exécution

Argument mapping (Jinja2)

{
  "version": "{{ nodes.build.result.version }}",
  "env": "{{ workflow.arguments.env }}"
}

Lancer un workflow

curl -X POST https://api.example.com/api/v1/workflows/<ID>/run \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"arguments": {"env": "production"}}'

Politiques d'échec

  • stop — arrêt au premier échec
  • continue — poursuit les branches indépendantes

Workers & labels

Assignation

Au moment du claim, le worker ne reçoit que les runs compatibles :

  • worker_id — worker précis (option job)
  • worker_group_id — membre d'un groupe
  • worker_labels — labels requis (ex. {"ansible": "true"})

Créer un worker via API

curl -X POST https://api.example.com/api/v1/workers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "worker-paris",
    "labels": {"docker": "true", "region": "paris"},
    "max_concurrency": 5
  }'
# Retourne le token rf_wkr_... (affiché une seule fois)

CLI worker

runflow-worker version
runflow-worker register --server URL --registration-token rf_reg_...
runflow-worker start

Secrets & credentials

Secrets

Valeurs chiffrées AES-256-GCM. Injectées comme variables d'environnement dans les conteneurs de job. Jamais affichées en clair dans l'UI ou les logs (redaction automatique).

curl -X POST https://api.example.com/api/v1/secrets \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name": "API_KEY", "value": "sk-...", "scope": "organization"}'

Credentials

Clés SSH, tokens, etc. Référencez-les dans un job via credential_refs. Montés dans le workspace au moment de l'exécution.

RUNFLOW_MASTER_KEY doit être identique sur toutes les instances API et sauvegardée de manière sécurisée. Sans elle, les secrets stockés sont irrécupérables.

Assistant IA

Disponible dans l'onglet Code d'un job (panneau Ask AI).

Configuration

  1. Créer un provider AI via API (POST /api/v1/ai/providers)
  2. Types supportés : OpenAI-compatible, Ollama
  3. Définir RUNFLOW_MASTER_KEY pour chiffrer les clés API des providers

Flux

  1. Sélectionner un provider et décrire la modification souhaitée
  2. L'IA propose des changements multi-fichiers (diff)
  3. Cliquer Apply pour écrire les fichiers

Sauvegardes recommandées

  • data/postgres/ — dump régulier via pg_dump
  • data/runflow/ — fichiers jobs et résultats
  • .env — hors Git, sauvegardé de manière sécurisée

Sécurité

  • Mots de passe : Argon2id
  • API Keys / Worker tokens : SHA-256, préfixe public
  • JWT : HS256, expiration configurable
  • RBAC : owner, admin, operator, viewer
  • Scopes API granulaires
  • Redaction automatique des secrets dans les logs
  • Anti path-traversal sur les fichiers jobs
  • Conteneurs de job isolés (pas de socket Docker)
  • Templates Jinja2 en sandbox
  • Chiffrement AES-256-GCM des secrets (RUNFLOW_MASTER_KEY)

Dépannage

Le worker reste offline

  • Vérifier RUNFLOW_API_URL (HTTPS accessible depuis l'agent)
  • Vérifier data/worker-<name>/worker.env contient un token rf_wkr_...
  • Consulter les logs : docker compose -f deploy/docker-compose.agents.yml logs worker
  • Vérifier que les images runner existent : docker images | grep runflow

Les runs restent en queued

  • Aucun worker online ou labels incompatibles
  • Worker à capacité max (max_concurrency)
  • Vérifier l'assignation : labels du job vs labels du worker

Erreur CORS dans le navigateur

  • CORS_ORIGINS doit correspondre exactement à l'URL web (avec https://)
  • NEXT_PUBLIC_API_URL doit pointer vers l'URL API publique
  • Rebuild l'image web après modification

Secrets illisibles / EncryptionError

  • RUNFLOW_MASTER_KEY manquante ou modifiée après stockage de secrets
  • La clé doit être identique à chaque redémarrage API

Certificat TLS Traefik

  • Vérifier que les DNS pointent vers le serveur
  • Vérifier TRAEFIK_CERT_RESOLVER correspond à votre config Traefik
  • Le conteneur doit être sur le réseau proxy

Job Docker échoue immédiatement

  • Image runner absente sur l'agent : relancer ./deploy/build-runners.sh
  • Permission Docker socket : l'utilisateur du conteneur worker doit accéder au socket
  • Consulter les logs du run dans l'UI ou via API
